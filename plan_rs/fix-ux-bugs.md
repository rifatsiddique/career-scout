## Bug A: File paths not clickable (P1 not being followed)

### Symptom

After `cv` mode finishes, the output reads:
```
I have successfully compiled the tailored Lutron PDF CV to output/cv-siddique-lutron-2026-05-22.pdf
```
and the Next Steps block shows:
```
### Next Steps
1. Review the PDF CV: Open cv-siddique-lutron-2026-05-22.pdf ...
2. Study the Prep Guide: Read through lutron-senior-hardware-engineering-manager.md ...
```

Relative paths; the `### Next Steps` header format differs from the P2 template.

### Root cause

P1 says "print absolute paths with `file://` prefix" but never tells the AI *how to
determine the project root*. The AI cannot guess the working directory, so it falls back
to relative paths. The `### Next Steps` format drift happens because the P2 template
says `What to do next:` but cv.md's old output block predates the P2 standard — the
AI has two competing patterns and picks the wrong one.

### Fix

**1. `modes/_shared.md` P1** — Add an explicit "how to resolve the project root" block:

> Before printing any file path, run a shell command to get the working directory.
> On Windows: `cd` (no arguments) prints the absolute path.
> On macOS/Linux: `pwd`.
> Cache the result as `PROJECT_ROOT`. Every file path in terminal output uses:
> `file:///{PROJECT_ROOT}/{relative-path}`
> (forward slashes, no trailing slash on PROJECT_ROOT, no leading slash on relative-path).

**2. `modes/cv.md` post-generation output** — Replace the current block with an explicit
instruction that uses the resolved PROJECT_ROOT and locks the P2 format:

```
Resolve PROJECT_ROOT via shell (cd / pwd). Then print:

📂 CV: file:///{PROJECT_ROOT}/output/{filename}.pdf

What to do next:
  1. Open the PDF above and review before submitting
  2. Submit your application — then update status → pipeline
  3. Once they schedule an interview → interview-prep {company-slug}

[P3 — only if composite ≥ 80]:
💡 After submitting and landing an interview, run: interview-prep {company-slug}
```

**3. `modes/interview-prep.md` terminal output** — Same fix: add the PROJECT_ROOT
resolution instruction before the P1 path print.

**4. Delete the old "Tell the user" block** that still exists above the new P1/P2 output
in cv.md (it's the pre-Phase-4 message that says "CV generated: output/{filename}.pdf"
in plain text). Having two output blocks causes the AI to pick whichever it sees first.

### Test

After fix: run `cv` mode and verify that:
- The terminal output shows `file:///C:/Work/Git-Python/career-scout/output/cv-...pdf`
  as a hyperlink in Windows Terminal / VS Code
- The "What to do next:" header appears (not "### Next Steps")
- No relative paths appear anywhere in the terminal output

## Bug B: Classic-professional template header — misleading title + missing contact fields

### Symptom

Line 2 of the CV header shows `{{CURRENT_TITLE}}` which is filled with the **target role
title** (e.g., "Senior Hardware Engineering Manager"). This is the role the candidate is
*applying to*, not who they are. A recruiter reading the CV sees a title that doesn't match
the candidate's actual current level — confusing, and potentially off-putting for slightly
different roles.

Line 3 (contact row) is missing phone number rendering (field exists but may be blank) and
has no immigration/work authorization status, which is frequently the first thing US
recruiters check for non-citizen candidates.

### Fix

**1. Replace `{{CURRENT_TITLE}}` with `{{HEADLINE}}`** in the template HTML.

`{{HEADLINE}}` is a short professional tag line generated by the drafter from the
candidate's actual most-recent role/expertise (NOT the target role title). It tells the
reader who the candidate is today, not what job they want.

Examples:
- `Power Electronics Engineer — LLC, PSFB, DAB | 8+ years`
- `Senior Hardware Engineering Manager | Power Conversion & Magnetics`
- `AI Platform Engineer — MLOps & Agentic Systems`

Format: `{Role/Domain} | {1-2 key differentiators}` — one line, no period.

The drafter derives this from `cv.md` most recent role + `_profile.md` archetype's
"what they buy" column. It does NOT read from the JD. If the user has a `headline:`
field in `profile.yml`, use that instead and skip derivation.

**2. Add `{{WORK_AUTH}}` to the contact row** (after LOCATION or PHONE).

Source: `profile.yml → visa_status`. Examples: "US Citizen", "Green Card Holder",
"H-1B (employer sponsorship required)", "Authorized to work in UK".

Skip (render nothing, remove separator) if `visa_status` is empty.

**3. Ensure `{{PHONE}}` is rendered or hidden cleanly.**

If `profile.yml → phone` is empty: remove the `{{PHONE}}` span AND its following
separator from the contact row. Don't leave a dangling `|` separator.

### Changes required

| File | Change |
|------|--------|
| `templates/cv/classic-professional.html` | Replace `{{CURRENT_TITLE}}` → `{{HEADLINE}}`; add `{{WORK_AUTH}}` to contact row with conditional separator; conditional `{{PHONE}}` separator |
| `config/profile.yml` | Add `headline: ""` optional field with comment (if user fills it, drafter uses it verbatim) |
| `modes/cv.md` | Update `{{CURRENT_TITLE}}` → `{{HEADLINE}}` in placeholder vocabulary table; add fill logic: "read profile.yml headline if set; else derive from cv.md most recent role + _profile.md archetype"; add `{{WORK_AUTH}}` to placeholder table sourced from `profile.yml → visa_status` |
| `README.md` | Update CSS variables table note if needed (no change expected) |

### CSS for the headline (same `.header-title` class is fine)

The italic small-font `.header-title` style works well for a professional tag line —
no CSS changes needed. Verify it still looks right with a longer headline like
"Power Electronics Engineer — LLC, PSFB, DAB | 8+ years".
